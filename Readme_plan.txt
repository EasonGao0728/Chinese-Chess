**理想结构**
你这个中国象棋项目，最稳的结构应该是 5 层：

1. 入口层 main  
只做一件事：创建 Game 对象并启动游戏。

2. 游戏控制层 Game  
负责回合切换、接收输入、调用规则判断、执行走子、判断胜负、保存历史。

3. 规则层 RuleChecker  
负责“全局规则”，例如：
红黑轮流走
不能吃己方棋子
走后不能让自己被将军
将帅不能照面
判断将军、将死、困毙

4. 棋盘层 Board  
只负责“棋局状态”与“状态修改”，例如：
某格有没有棋子
某格是什么棋子
两点之间隔了几个子
执行一步棋
撤销一步棋
查找某方将的位置

5. 棋子层 Chess + 各子类  
只负责“这个棋子本身的走法模式”：
车走直线
马走日且别马腿
象走田且不过河
炮平时走直线、吃子隔一子
兵过河前后规则不同

这 5 层里，最重要的分工原则是：
棋子类只判断“这颗子按自身规则能不能这么走”；
RuleChecker 再判断“这步在整盘棋里是否合法”。

**总路线**
你可以把后续工作拆成 8 步。每一步我都写成“要做什么、为什么现在做、做到什么算完成、常见坑”。

1. 第一步：先统一数据模型
目标：明确整个项目里最基础的几个对象到底是什么，以及谁负责什么。

你现在先定死这几个概念：
棋盘坐标 pos
棋子基类 Chess
七种棋子子类
棋盘 Board
一步棋 Move
规则检查 RuleChecker
游戏流程 Game
控制台界面 ConsoleUI

这一步你要做的事：
把每个类的职责先写出来，哪怕先写在笔记里也行。
规定一个原则：棋子类只管自身走法，Board 只管保存棋局，RuleChecker 只管全局合法性，Game 只管流程控制。

为什么先做这个：
因为你现在的代码已经开始有“职责混在一起”的迹象，比如将帅照面这种全局规则已经写进了 King.cpp。如果不先分清职责，后面会反复返工。

完成标准：
你能一句话说清每个类的责任边界。

常见坑：
把“吃子判断、照将判断、轮流走棋判断”全部塞进棋子类里。

2. 第二步：重构 Chess 基类
目标：让所有棋子都能通过统一接口判断走法。

你现在的 Chess.h 还是普通数据类，下一步应改成“抽象基类”。

你要做的事：
给 Chess 保留公共属性：
位置
阵营
存活状态
类型

然后补这些公共函数：
获取和设置位置
获取阵营
获取是否存活
获取类型
虚析构函数
纯虚函数：判断该棋子按自身规则能否从起点走到终点

推荐你把这个纯虚函数名字统一成一种，例如：
canMovePattern
或者
isLegalMovePattern

这一层只判断“这个棋子本身是不是这么走”，先不要判断“走完会不会被将军”。

为什么现在做：
因为这是整个多态设计的入口。如果这一步不改，后面 Board 里存子类对象就没有意义。

完成标准：
你能通过 Chess 指针调用不同子类的走法判断函数。

常见坑：
忘记写虚析构函数。
把全局规则写进纯虚函数的实现里。
基类 type 写得不统一，后面字符串比较容易出问题。

3. 第三步：重构 Board 存储方式
目标：让棋盘真正保存“各种具体棋子”，而不是被切成基类副本。

你现在的 Board.h 里是 Chess pieces[32]，这个设计最大的问题就是对象切片。

你要做的事：
把 Board 的存储改成下面两种方案之一。

方案 A：9×10 棋盘格存指针
每个格子要么为空，要么指向一个 Chess 对象。

方案 B：Board 统一持有 32 个棋子对象，再额外维护位置查询
比如用一个容器保存所有棋子，再通过坐标查谁在这个位置。

对你这个作业来说，我更建议方案 A，思路直观。

Board 至少要补这些函数：
初始化棋局
判断坐标是否越界
判断格子是否为空
获取某格棋子
统计两点之间有几个棋子
把棋子从起点移动到终点
执行吃子
查找某方将的位置

为什么现在做：
因为所有棋子规则几乎都依赖 Board 查询能力。车、炮、马、象都要依赖路径或中间格判断。

完成标准：
Board 能正确回答这几个问题：
某格有没有子
某格是谁
两点之间隔了几个子
某方的将在哪

常见坑：
返回局部对象引用。
让 Board 同时承担规则判断和游戏流程控制。
坐标轴方向前后不统一。

4. 第四步：把 7 个棋子的基础走法全部整理干净
目标：先只做“棋子自身规则”，不碰全局规则。

你现在这部分已经写了一半，但要重新梳理一遍，统一接口、统一边界判断、统一返回逻辑。

你要逐个完成：

车
直线走
路径中间不能有子

马
走日
检查马腿

象
走田
不能过河
检查象眼

士
斜走一步
不能出九宫

将
上下左右一步
不能出九宫
先不要单独在这里处理完整的照面逻辑，留给 RuleChecker 更合适

炮
不吃子时，路径中间必须没有子
吃子时，路径中间必须恰好有一个子

兵
红兵和黑兵方向不同
过河前只能前进
过河后可以左右，不能后退

为什么这一步单独做：
因为这里是“局部规则”，最容易验证，也最适合先收敛。

完成标准：
对于任意一个棋子，你都能只靠它自己的走法函数判断“形状上是否合法”。

常见坑：
炮判断中间是否隔一个子时把起点或终点也算进去。
车、炮路径检查时循环从起点开始，导致起点自己被算成阻挡。
兵的红黑方向写反。
Pawn source.cpp 这一类代码你后面一定要重点复核，因为兵的方向最容易错。

5. 第五步：建立 Move 类，让一步棋变成完整对象
目标：后面所有执行、撤销、历史记录、悔棋、将军判断都围绕 Move 来做。

你现在的 Move.h 方向是对的，但还不够完整。

你要让 Move 至少记录：
起点
终点
行棋方
移动的棋子
被吃的棋子
是否吃子
第几步

如果你准备做悔棋，还建议额外记录：
移动前的原位置
被吃棋子的原状态

为什么现在做：
因为没有统一的 Move 对象，Game、Board、RuleChecker 之间的信息传递会很乱。

完成标准：
一次走子行为能被完整描述，并且可以用于日志和撤销。

常见坑：
在 Move 里存对临时对象的引用。
只记录坐标，不记录被吃子的状态，后面撤销会很难做。

6. 第六步：新建 RuleChecker，集中处理全局规则
目标：把“整盘棋层面的合法性”从棋子类里剥离出来。

这是整个项目里最关键的一步。

RuleChecker 建议先做这几个函数：
判断一步棋是否整体合法
判断某一方是否被将军
判断走完后是否导致自己仍然被将军
判断将帅是否照面
判断某一方是否还有合法走法
判断将死或困毙

建议你按这个顺序写：

先写：
是否轮到该方走
起点有没有棋
终点是否是己方棋
棋子自身走法是否成立

再写：
临时执行这一步
检查自己是否被将军
检查将帅是否照面
不合法则回退

再写：
检查对方是否被将军

为什么这一步要晚于棋子规则：
因为 RuleChecker 依赖两个基础：
Board 查询能力够完整
各棋子基础走法已经可信

完成标准：
Game 在尝试走棋时，只需要问 RuleChecker 一句话：这步能不能走。

常见坑：
把“自己被将军”的判断忘了做。
只检查目标点是否合法，不检查走完后的整局状态。
照面规则只在 King 里判断一次，导致其他棋子挪开中间遮挡后没有被拦住。

7. 第七步：完成 Board 的执行与撤销
目标：让棋局状态真正能前进、也能回退。

你要给 Board 写两组关键操作：

执行一步棋：
把起点棋子移到终点
若终点有敌子则吃掉
更新棋子位置和存活状态

撤销一步棋：
把行棋子放回原位
恢复被吃棋子
恢复存活状态

为什么这一步单列出来：
因为 RuleChecker 在判断“走完会不会被将军”时，最简单的方法就是：
先模拟执行
再检查局面
然后撤销

完成标准：
你能放心地“试走一步再撤销”，而不会把棋盘搞乱。

常见坑：
吃子后只是从棋盘上删掉，但没改 alive。
撤销时只恢复位置，没恢复被吃子的存活状态。

8. 第八步：写 Game 主循环，把整局跑起来
目标：让整个游戏真正可玩。

Game 建议先有这些成员：
一个 Board
一个 RuleChecker
当前轮到谁
步数计数
历史 Move 列表
游戏是否结束

主循环流程固定成这样：
显示棋盘
提示轮到哪方
读入起点终点
构造 Move
调用 RuleChecker 判断
如果合法，Board 执行，保存历史，切换回合
如果不合法，提示原因，重新输入
每走一步后判断将军、将死、困毙、胜负

为什么这一步放这么后：
因为它是装配层。底下模块不稳，Game 写了也只是不断打补丁。

完成标准：
你可以在控制台里连续走多步，直到一方获胜。

常见坑：
Game 里直接写一大堆规则细节。
UI 输入解析失败时没有重试逻辑。
切换回合写在非法走子之前。

9. 第九步：补 ConsoleUI，专门负责输入输出
目标：让界面层干净，不污染游戏逻辑。

ConsoleUI 要做的事很简单：
打印当前棋盘
提示谁走
读玩家输入
把输入解析成坐标
显示错误信息
显示将军或获胜信息

建议你先用最简单的坐标输入格式，比如：
x1 y1 x2 y2

为什么要单独分层：
因为后面即使你想换成图形界面，Game 和 RuleChecker 也不用重写。

完成标准：
控制台界面只负责“看”和“输”，不负责“判”。

常见坑：
在 ConsoleUI 里直接调用棋子规则函数。
把棋盘打印逻辑混进 Board 里。

10. 第十步：系统测试，不要边玩边猜
目标：用固定样例验证核心规则。

你至少要设计这几类测试：

基础移动测试
车、马、象、士、将、炮、兵都能正确合法移动

非法移动测试
越界
吃己方
不符合走法

特殊规则测试
马腿
象眼
炮架
兵过河
将帅照面

局面规则测试
被将军后不能乱走
必须解将
将死判定
困毙判定

为什么必须专门测：
中国象棋规则多，靠手动玩几步很容易漏 bug。

完成标准：
每条核心规则至少有一个明确样例验证过。

常见坑：
只测“能走”的情况，不测“不能走”的情况。
只测开局，不测残局。
